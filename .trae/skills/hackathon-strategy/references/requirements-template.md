## クイック要件定義

15分で完了する最小限の要件定義。アイデアが明確な場合に使用。

```markdown
# [プロジェクト名]

## ピッチ (1文)
[WHO] が [WHAT] できる [HOW] を使った [TYPE]

## 解決する課題
[具体的な課題を1-2文で]

## ターゲットユーザー
[誰が使うのかを1文で]

## コア機能 (最大3つ)
1. [ ] [機能名]: [何ができるか] (見積: Xh)
2. [ ] [機能名]: [何ができるか] (見積: Xh)
3. [ ] [機能名]: [何ができるか] (見積: Xh)

## Gemini 3 活用ポイント
- [どの機能でGemini 3を使うか]
- [なぜGemini 3が最適か]

## デモハイライト
[2分間のデモで最も見せたい瞬間]
```

## スコープ管理マトリクス

### MoSCoW法によるスコープ分類

```
Must Have (MVP):
  - 制限時間内に必ず完成させる
  - これがないとデモが成立しない
  - 最大3機能に制限

Should Have (Stretch):
  - 時間があれば追加
  - デモの完成度を上げる
  - 2-3機能まで

Could Have (Nice-to-have):
  - 完全にオプション
  - プレゼンで「将来は〇〇も」と言及可能

Won't Have (Out of scope):
  - 明示的に除外
  - チームが迷わないように記載
```

### 時間配分の目安

```
全体時間を 100% とした場合:
  MVP実装:     50% (最重要)
  統合テスト:   15%
  デモ準備:     15%
  Stretch機能:  10%
  バッファ:     10%
```

## リスク評価チェックリスト

### 開始前に確認

- [ ] Gemini 3 APIのアクセス確認済み
- [ ] チーム全員の開発環境セットアップ済み
- [ ] 使用するライブラリ/フレームワークの動作確認済み
- [ ] デモ用のテストデータ準備済み
- [ ] Wi-Fi / ネットワーク接続確認済み

### リスク一覧

| リスク | 影響度 | 発生確率 | 対策 |
|--------|--------|----------|------|
| API レート制限 | 高 | 中 | キャッシュ、フォールバック |
| ネットワーク不安定 | 高 | 中 | オフラインモード準備 |
| スコープクリープ | 高 | 高 | MVP厳守、タイマー設定 |
| チーム内意見不一致 | 中 | 中 | 最初の30分で合意 |
| デモ環境の問題 | 高 | 低 | 事前テスト、バックアップ |

### 中間チェックポイント

```
25% 時点: コア機能のAPIコールが動作するか?
  → No: スコープ縮小を検討

50% 時点: MVPの全機能が結合して動作するか?
  → No: 最も重要な1機能に集中

75% 時点: デモシナリオ通りに動作するか?
  → No: デモ可能な部分でプレゼンを組み直す
```
